# stock-select-rs Chart Bridge Design

## Goal

Keep the Rust CLI workflow compatible with Python `stock-select` while the native Rust implementation is expanded. The next chart phase should preserve user-facing artifacts and runtime layout first, then optimize internals after parity tests are in place.

## Current State

`stock-select-rs run` is hybrid:

1. Rust `screen` reads PostgreSQL, prepares indicators, writes `candidates/<pick_date>.<method>.json`.
2. Rust calls Python `stock-select chart` through `uv run`.
3. Rust calls Python `stock-select review` through `uv run`.

For `2026-05-25 b1`, this hybrid path has parity against the existing Python runtime:

- screen candidate set: `104/104`
- review summary: `reviewed=104 recommendations=3`
- chart output count: `104` daily PNG files

Candidate order is part of the contract. Rust now writes candidates in `code` ascending order to match Python, because Python chart and review consume candidate order directly.

## Chosen Approach

Use `Rust Command -> uv run python bridge` for the chart phase.

The chart command remains:

```text
uv run stock-select chart --method METHOD --pick-date DATE --runtime-root ROOT
```

Rust owns the orchestration and stage timing. Python owns chart rendering until review is ported and chart parity has a dedicated visual test gate.

## Alternatives Considered

### Keep Python CLI Bridge

Pros:

- Lowest risk because current PNGs are generated by the same Python code as the golden workflow.
- Keeps `mplfinance`, matplotlib fonts, panel layout, and file naming unchanged.
- Easy to validate with file count, filename set, and image smoke tests.
- Avoids `pyo3` packaging and Python interpreter lifecycle issues in the Rust binary.

Cons:

- Requires the Python project and `uv` environment at runtime.
- Process startup overhead remains, though it is small compared with rendering 100+ charts.
- Chart performance remains bounded by Python chart implementation.

### Embed Python With pyo3

Pros:

- One Rust process can call Python functions without repeated CLI startup.
- Gives Rust tighter control over errors and inputs.

Cons:

- Adds native build and packaging complexity.
- Still depends on Python, matplotlib, mplfinance, font configuration, and Python runtime state.
- Makes failures harder to isolate than a subprocess boundary.
- Does not materially solve the main chart bottleneck before chart internals are optimized.

### Native Rust Chart Rendering

Pros:

- Long-term fastest and most deployable option if implemented well.
- Removes Python/matplotlib dependency.

Cons:

- Highest visual parity risk.
- Requires recreating mplfinance candle layout, overlays, MACD panel, colors, scaling, labels, and image sizing.
- Should not start until review parity and chart smoke gates are stable.

## Runtime Contract

For end-of-day mode, the chart bridge must read:

```text
<runtime_root>/candidates/<pick_date>.<method>.json
```

and write:

```text
<runtime_root>/charts/<pick_date>.<method>/<code>_day.png
```

The chart stage must not rewrite candidates. It may read PostgreSQL through Python chart code exactly as the current Python CLI does.

## Environment Profile Boundary

Environment profile is not implemented natively in Rust yet.

Current Rust behavior:

- `stock-select-rs run` accepts `--environment-state` and `--environment-reason`.
- Rust forwards those values only to the Python `review` stage.
- Rust `screen` does not call Python `ensure_market_environment`.
- Rust does not evaluate market environment state.
- Rust does not implement `MethodEnvironmentProfile` weights, thresholds, `llm_focus`, or b1/b2 environment-specific review gates.

Current Python behavior:

- Python `screen` can call `ensure_market_environment` when no manual environment is supplied.
- Python `review` resolves environment context through `resolve_market_environment`.
- Python `review` applies `get_method_environment_profile` for `b1`, `b2`, and `left_peak`.
- Python `review` writes `environment_state`, `environment_reason`, and `environment_llm_focus` into `llm_review_tasks.json`.

While review is bridged to Python, environment profile behavior remains available through Python review. Full Rust replacement requires a separate review/environment port.

### Golden Gate

Compare Rust hybrid artifacts against the existing Python runtime:

```bash
python3 scripts/compare_screen.py \
  --python-root ~/.agents/skills/stock-select/runtime \
  --rust-root /tmp/stock-select-rs-chart-gate \
  --pick-date 2026-05-25 \
  --method b1

python3 scripts/compare_review.py \
  --python-root ~/.agents/skills/stock-select/runtime \
  --rust-root /tmp/stock-select-rs-chart-gate \
  --pick-date 2026-05-25 \
  --method b1
```

Expected:

- screen comparison passes
- review comparison passes

### Visual Smoke Gate

Add a future script that checks:

- PNG filename set equals candidate code set.
- PNG dimensions match the Python chart output dimensions.
- PNG files decode successfully.
- A small set of reference charts has non-blank candle area and MACD panel pixels.

This is a smoke gate, not pixel-perfect comparison. Pixel-perfect output is too brittle across matplotlib/font/rendering environments.

## Next Implementation Work

1. Keep the current subprocess Python chart bridge.
2. Add a chart smoke comparison script before replacing chart internals.
3. Avoid `pyo3` until there is evidence that CLI process startup is a meaningful bottleneck.
4. Prioritize native Rust review/environment profile port before native chart rendering, because review parity controls final selection results.
