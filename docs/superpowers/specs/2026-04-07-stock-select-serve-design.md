# Stock Select Serve Command Design

## Goal

Add a standalone CLI command that serves the HTML charts generated by `stock-select chart` so an agent or human can open them in a browser.

## Scope

Included:

- a new `serve` CLI command
- static HTTP serving for one pick date's chart directory
- clear startup output with the local base URL
- skill documentation updates so agents know when and how to use the command

Excluded:

- changing `chart` or `run` behavior
- regenerating charts as part of `serve`
- adding a chart index page in this change
- any remote hosting or authentication

## Current State

- `chart` writes files to `~/.agents/skills/stock-select/runtime/charts/<pick_date>/`
- each candidate chart is written as `<code>_day.html`
- the `stock-select` skill explains chart generation, but not how to expose those HTML files over HTTP

This means downstream agents can produce chart HTML, but they do not have a standard CLI entrypoint for browser access.

## Chosen Approach

Add a new command:

```bash
stock-select serve --pick-date YYYY-MM-DD
```

Optional flags:

- `--runtime-root PATH`
- `--host TEXT` with default `127.0.0.1`
- `--port INTEGER` with default `8000`

The command serves the existing directory:

```text
<runtime_root>/charts/<pick_date>/
```

If the directory does not exist, the command fails with a message telling the caller to run `chart` first.

## Command Behavior

### Inputs

- required `--pick-date`
- optional `--runtime-root`
- optional `--host`
- optional `--port`

No method flag is needed because the command only exposes already-rendered chart files and does not participate in strategy selection.

### Outputs

On success, the command:

- starts a local static file server rooted at `charts/<pick_date>/`
- emits the base URL to stdout, for example `http://127.0.0.1:8000/`
- keeps running until interrupted

The caller can then open chart files such as:

```text
http://127.0.0.1:8000/000001.SZ_day.html
```

### Error Handling

- if `charts/<pick_date>/` does not exist, fail fast with a user-facing error
- if the port is unavailable, surface the server error rather than silently choosing another port

## Implementation Notes

- use the standard library HTTP server rather than adding a new dependency
- isolate server setup behind a small helper so the CLI test surface stays narrow
- keep the command read-only with respect to runtime outputs

## Skill Update

Update the `stock-select` skill so agents know:

- `chart` generates HTML files but does not make them browser-accessible by itself
- `serve --pick-date ...` is the standard way to expose those files over HTTP
- `serve` must be run after `chart`
- the served directory maps directly to `runtime/charts/<pick_date>/`

## Testing

Add CLI tests that cover:

- missing chart directory returns an error
- successful invocation starts the server helper with the resolved directory, host, and port
- stdout includes the base URL

The actual long-running HTTP loop should be stubbed in tests.
